Exercice 1

Gestion des utilisateurs, privileges, roles et profils

Securite logique Oracle: qui peut se connecter, ce qu'il peut faire, dans quel espace, et quelles limites s'appliquent.

1. Utilisateurs

Comment ca marche: Un utilisateur Oracle est une identite + un schema (espace d'objets). Oracle verifie l'identifiant et le mot de passe avant d'ouvrir la session. Sans le privilege CREATE SESSION, l'utilisateur ne peut pas se connecter meme si le compte existe. Sans quota sur un tablespace, il ne peut pas creer d'objets permanents dans ce tablespace. L'authentification peut etre geree par Oracle (IDENTIFIED BY) ou par l'OS (IDENTIFIED EXTERNALLY).
CREATE USER — creation complete 
CREATE USER app_user IDENTIFIED BY mon_password
  DEFAULT TABLESPACE users
  TEMPORARY TABLESPACE temp
  QUOTA 50M ON users
  ACCOUNT UNLOCK;

-- Puis accorder le droit de connexion (OBLIGATOIRE)
GRANT CREATE SESSION TO app_user;

-- Authentification par l'OS (pas de mot de passe Oracle)
CREATE USER user_os IDENTIFIED EXTERNALLY;
ALTER USER — toutes les modifications 
ALTER USER app_user IDENTIFIED BY nouveau_password;   -- Changer mot de passe
ALTER USER app_user ACCOUNT LOCK;                     -- Verrouiller
ALTER USER app_user ACCOUNT UNLOCK;                   -- Deverrouiller
ALTER USER app_user PASSWORD EXPIRE;                  -- Forcer changement
ALTER USER app_user QUOTA 100M ON users;              -- Modifier quota
ALTER USER app_user QUOTA UNLIMITED ON users;         -- Quota illimite
ALTER USER app_user QUOTA 0 ON users;                 -- Supprimer quota
ALTER USER app_user DEFAULT TABLESPACE data_tbs;      -- Changer tablespace
ALTER USER app_user PROFILE profil_app;               -- Affecter un profil
DROP USER 
DROP USER app_user;           -- Echoue si l'utilisateur possede des objets
DROP USER app_user CASCADE;   -- Supprime l'utilisateur ET tout son schema
Quand ca ne marche pas:
  • ORA-01045 — connexion refusee: CREATE SESSION manquant → GRANT CREATE SESSION TO user
  • ORA-01536 — quota depasse sur tablespace → ALTER USER ... QUOTA UNLIMITED ON tbs
  • ORA-28000 — compte verrouille (trop d'echecs) → ALTER USER ... ACCOUNT UNLOCK
  • ORA-28001 — mot de passe expire → ALTER USER ... IDENTIFIED BY nouveau_mdp
  • DROP refuse — schema non vide → ajouter CASCADE
Vues de verification 
SELECT username, account_status, default_tablespace, profile FROM dba_users;
SELECT username, tablespace_name, bytes, max_bytes FROM dba_ts_quotas WHERE username = 'APP_USER';

2. Privileges

Comment ca marche: Deux types totalement differents: Privilege systeme = autorise une action generale sur la base (creer une table dans n'importe quel schema). Privilege objet = autorise une action sur un objet precis d'un autre schema (SELECT sur la table EMP de SCOTT). On donne avec GRANT, on retire avec REVOKE. Pour transmettre un privilege systeme → WITH ADMIN OPTION. Pour transmettre un privilege objet → WITH GRANT OPTION.
TypePorteeOption de transmissionExemples
Privilege systeme Action globale sur la base WITH ADMIN OPTION CREATE SESSION, CREATE TABLE, ALTER SYSTEM
Privilege objet Action sur un objet precis WITH GRANT OPTION SELECT ON scott.emp, EXECUTE ON proc
GRANT — privileges systeme 
GRANT CREATE SESSION TO app_user;
GRANT CREATE TABLE TO app_user;
GRANT CREATE VIEW, CREATE SEQUENCE, CREATE PROCEDURE TO app_user;
GRANT UNLIMITED TABLESPACE TO app_user;
-- Avec droit de transmettre le privilege
GRANT CREATE TABLE TO app_user WITH ADMIN OPTION;
GRANT — privileges objet 
GRANT SELECT, INSERT, UPDATE, DELETE ON scott.emp TO app_user;
-- Privilege sur colonnes specifiques seulement
GRANT UPDATE (sal, comm) ON scott.emp TO app_user;
-- Avec droit de transmettre le privilege
GRANT SELECT ON scott.emp TO app_user WITH GRANT OPTION;
-- Donner a tous les utilisateurs
GRANT SELECT ON scott.emp TO PUBLIC;
REVOKE 
REVOKE CREATE TABLE FROM app_user;
REVOKE SELECT ON scott.emp FROM app_user;
REVOKE INSERT, UPDATE, DELETE ON scott.emp FROM app_user;
Vues de verification 
SELECT grantee, privilege, admin_option FROM dba_sys_privs WHERE grantee = 'APP_USER';
SELECT grantee, owner, table_name, privilege FROM dba_tab_privs WHERE grantee = 'APP_USER';
SELECT * FROM session_privs;   -- Privileges actifs dans la session courante

3. Roles

Comment ca marche: Un role est un conteneur nomme de privileges. Au lieu de donner 10 privileges individuellement a 50 utilisateurs, on cree un role, on y met les privileges, puis on donne le role aux utilisateurs. Un seul REVOKE du role retire tous ses privileges a l'utilisateur. Oracle propose des roles predefinies: CONNECT, RESOURCE, DBA, SYSBACKUP.
Cycle complet d'un role 
-- Creer le role
CREATE ROLE role_compta;

-- Donner des privileges au role
GRANT CREATE SESSION TO role_compta;
GRANT SELECT, INSERT, UPDATE ON finance.factures TO role_compta;

-- Donner le role a un utilisateur
GRANT role_compta TO app_user;
GRANT role_compta TO app_user WITH ADMIN OPTION;  -- Peut transmettre le role

-- Retirer le role d'un utilisateur
REVOKE role_compta FROM app_user;

-- Supprimer le role
DROP ROLE role_compta;
Role predefinedContenuQuand l'utiliser
CONNECTCREATE SESSION principalementConnexion basique
RESOURCECREATE TABLE, SEQUENCE, PROCEDURE...Developpeur
DBAPresque tous les privileges systemeAdministrateur (a limiter)
SYSBACKUPBackup et recovery sans acces aux donneesRMAN — moindre privilege
Exam — distinction cle: SYSBACKUP est un ROLE (pas un privilege systeme). Il est prefere au role DBA pour RMAN: l'administrateur RMAN n'a pas besoin d'acceder aux donnees applicatives.
WITH GRANT OPTION = transmettre un privilege objet. WITH ADMIN OPTION = transmettre un privilege systeme ou un role.
Vues de verification 
SELECT grantee, granted_role, default_role, admin_option FROM dba_role_privs WHERE grantee = 'APP_USER';
SELECT * FROM session_roles;   -- Roles actifs dans la session courante

4. Profils

Comment ca marche: Un profil regroupe des limites de deux categories: Limites de mots de passe (toujours actives) et limites de ressources (actives seulement si RESOURCE_LIMIT = TRUE au niveau systeme). Chaque utilisateur a exactement un profil. Si aucun n'est affecte, le profil DEFAULT s'applique.
Parametre mot de passeCe que ca fait
FAILED_LOGIN_ATTEMPTSNb max de tentatives echouees avant verrouillage du compte
PASSWORD_LOCK_TIMEDuree du verrouillage en jours (UNLIMITED = permanent jusqu'au DBA)
PASSWORD_LIFE_TIMEDuree de validite du mot de passe en jours
PASSWORD_GRACE_TIMEPeriode de grace apres expiration: connexion encore possible mais changement requis
PASSWORD_REUSE_TIMEDelai en jours avant reutilisation d'un ancien mot de passe
PASSWORD_REUSE_MAXNb de changements requis avant reutilisation d'un ancien mot de passe
Parametre ressourceCe que ca fait
SESSIONS_PER_USERNb max de sessions simultanees pour cet utilisateur
CONNECT_TIMEDuree max d'une session connectee (minutes)
IDLE_TIMEDuree max d'inactivite avant deconnexion automatique (minutes)
LOGICAL_READS_PER_SESSIONNb max de lectures logiques par session
CREATE PROFILE et affectation 
CREATE PROFILE profil_app LIMIT
  FAILED_LOGIN_ATTEMPTS  3
  PASSWORD_LOCK_TIME     1
  PASSWORD_LIFE_TIME     90
  PASSWORD_GRACE_TIME    7
  SESSIONS_PER_USER      2
  CONNECT_TIME           60
  IDLE_TIME              15;

-- Affecter le profil a un utilisateur
ALTER USER app_user PROFILE profil_app;

-- Activer les limites de ressources au niveau systeme (obligatoire)
ALTER SYSTEM SET resource_limit = TRUE;

-- Remettre le profil par defaut
ALTER USER app_user PROFILE DEFAULT;
Quand les limites de ressources n'ont aucun effet: Si RESOURCE_LIMIT = FALSE (defaut Oracle), les parametres SESSIONS_PER_USER, CONNECT_TIME, IDLE_TIME... sont completement ignores. Seules les limites de mots de passe sont toujours actives. → Solution: ALTER SYSTEM SET resource_limit = TRUE;

Difference PASSWORD_LOCK_TIME / PASSWORD_LIFE_TIME: LOCK_TIME = duree du verrouillage apres echecs. LIFE_TIME = duree de validite normale. Ce sont deux parametres independants.
Vues de verification 
SELECT profile, resource_name, limit FROM dba_profiles WHERE profile = 'PROFIL_APP';
SELECT username, profile FROM dba_users WHERE username = 'APP_USER';
Logique exercice 1: Securite Oracle = 4 niveaux. Utilisateur (identite + tablespace + quota) → Privilege (ce qu'il peut faire) → Role (regroupement de privileges) → Profil (limites et contraintes).
Exercice 2

RMAN — connexion, sauvegardes, recuperations, catalogue

RMAN (Recovery Manager) est l'outil Oracle integre pour toutes les operations de sauvegarde et recuperation. Il connait la structure interne de la base.

2a. Connexion RMAN

Comment ca marche: RMAN est un executable distinct de SQL*Plus. On le lance depuis le shell systeme avec la commande rman. Le mot-cle TARGET designe la base cible a sauvegarder. SYSBACKUP est le role recommande: il couvre backup et recovery sans donner acces aux donnees applicatives (principe du moindre privilege). Sans connexion valide a une base cible, RMAN ne peut rien faire.
Toutes les formes de connexion 
-- Connexion locale OS authentication (le plus courant)
rman TARGET /

-- Connexion locale avec le role SYSBACKUP (recommande)
rman TARGET "/ AS SYSBACKUP"

-- Connexion distante avec mot de passe
rman TARGET sys/password@mydb

-- Depuis l'invite RMAN (apres lancement sans argument)
rman
CONNECT TARGET /
CONNECT TARGET sys/password@mydb;

-- Connexion avec Recovery Catalog en plus
rman TARGET / CATALOG rcat_owner/password@catdb
Quand ca ne marche pas:
  • ORA-01031: SYSBACKUP pas accorde → GRANT SYSBACKUP TO user en SQL*Plus
  • Base inaccessible: la base n'est pas demarree → faire STARTUP en SQL*Plus d'abord
  • ORACLE_SID mal defini: export ORACLE_SID=orcl (Linux) ou set ORACLE_SID=orcl (Windows)
  • Erreur TNS: le service Oracle Net n'est pas configure → verifier le listener

2b. Vues et commandes de consultation RMAN

Comment ca marche: RMAN stocke l'historique de toutes ses operations dans le control file (ou dans le Recovery Catalog si configure). On peut interroger cet historique avec des commandes RMAN propres (LIST, REPORT, SHOW) ou avec des requetes SQL sur les vues dynamiques V$.
Commandes RMAN de consultation 
SHOW ALL;                              -- Configuration persistante RMAN
REPORT SCHEMA;                         -- Structure de la base (datafiles, tablespaces)
LIST BACKUP;                           -- Toutes les sauvegardes
LIST BACKUP SUMMARY;                   -- Resume des sauvegardes
LIST BACKUP OF DATABASE;               -- Sauvegardes de la base entiere
LIST BACKUP OF TABLESPACE users;       -- Sauvegardes d'un tablespace precis
LIST COPY;                             -- Copies image disponibles
LIST ARCHIVELOG ALL;                   -- Archived logs disponibles
REPORT OBSOLETE;                       -- Sauvegardes obsoletes selon la retention
Vues V$ importantes pour RMAN 
SELECT name, value FROM v$rman_configuration;            -- Config RMAN persistante
SELECT log_mode, flashback_on FROM v$database;            -- Mode archivelog + flashback
SELECT current_scn FROM v$database;                       -- SCN actuel
SELECT * FROM v$recovery_file_dest;                       -- FRA: chemin et taille
SELECT file_type, percent_space_used FROM v$recovery_area_usage;  -- Utilisation FRA
SELECT job_type, status, start_time FROM v$rman_backup_job_details ORDER BY start_time DESC;

2c. Types de sauvegarde

Prerequis pour les sauvegardes a chaud (base ouverte): La base doit etre en mode ARCHIVELOG. En NOARCHIVELOG, seule la sauvegarde a froid (base arretee) est possible. Pour passer en ARCHIVELOG: SHUTDOWN IMMEDIATE; STARTUP MOUNT; ALTER DATABASE ARCHIVELOG; ALTER DATABASE OPEN;

Whole backup (sauvegarde de toute la base)

Whole designe la portee de la sauvegarde: toute la base (tous les datafiles + control file). C'est la sauvegarde la plus courante. RMAN ne copie que les blocs utilises, pas les blocs vides. On peut y ajouter les archived logs avec PLUS ARCHIVELOG pour permettre une recuperation complete. 
-- Sauvegarde de toute la base
BACKUP DATABASE;

-- Avec archived logs (recommande — permet recuperation complete)
BACKUP DATABASE PLUS ARCHIVELOG;

-- Dans un bloc RUN avec canal dedie
RUN {
  ALLOCATE CHANNEL c1 DEVICE TYPE DISK FORMAT '/backup/%U';
  BACKUP DATABASE PLUS ARCHIVELOG;
  BACKUP CURRENT CONTROLFILE;
  DELETE OBSOLETE;
}

Full backup (sauvegarde non incrementale)

Full designe le niveau (level) de la sauvegarde: elle n'est pas incrementale. Un full backup copie tous les blocs utilises du fichier, sans reference a une sauvegarde precedente. Par defaut, BACKUP DATABASE est deja un full backup. Difference avec Incremental Level 0: un full backup ne peut pas servir de base pour les sauvegardes incrementales. Le level 0 peut. 
-- Full backup de la base (comportement par defaut)
BACKUP DATABASE;

-- Full backup d'un tablespace specifique
BACKUP TABLESPACE users;

-- Full backup en backup set (format compresse RMAN)
BACKUP AS BACKUPSET DATABASE;

-- Full backup en copie image (restauration plus rapide, mais plus d'espace)
BACKUP AS COPY DATABASE;

-- Full backup compresse
BACKUP AS COMPRESSED BACKUPSET DATABASE PLUS ARCHIVELOG;

Backup incremental

Le backup incremental ne sauvegarde que les blocs modifies depuis une sauvegarde precedente. Avantage: moins de temps et d'espace. Inconvenient: restauration plus longue (rejoue les niveaux).

Level 0: copie tous les blocs utilises (comme un full), mais sert de base de reference pour les level 1.
Level 1 differentielle: blocs modifies depuis le dernier incremental (level 0 OU level 1). Restore: level 0 + tous les level 1 differentiels successifs.
Level 1 cumulative: blocs modifies depuis le dernier level 0 uniquement. Restore: level 0 + le dernier level 1 cumulatif seulement (plus simple mais plus volumineux). 
-- Level 0: base de reference (point de depart de la strategie incrementale)
BACKUP INCREMENTAL LEVEL 0 DATABASE;

-- Level 1 differentielle (depuis le dernier incremental — level 0 ou 1)
BACKUP INCREMENTAL LEVEL 1 DATABASE;

-- Level 1 cumulative (depuis le dernier level 0 seulement)
BACKUP INCREMENTAL LEVEL 1 CUMULATIVE DATABASE;
TypeBlocs sauvegardesBase pour increm.?Restauration
Full backupTous les blocs utilisesNonDirecte
Increm. Level 0Tous les blocs utilisesOuiDirecte
Level 1 differentielleModifies depuis dernier increm.NonLevel 0 + tous les level 1
Level 1 cumulativeModifies depuis dernier level 0NonLevel 0 + dernier level 1
Quand les sauvegardes echouent:
  • Base en NOARCHIVELOG avec backup a chaud → passer en mode ARCHIVELOG d'abord
  • Espace insuffisant dans la FRA → ALTER SYSTEM SET db_recovery_file_dest_size = 50G
  • Control file non sauvegarde → activer CONFIGURE CONTROLFILE AUTOBACKUP ON
  • Permissions insuffisantes sur le repertoire → verifier les droits systeme du compte Oracle

2d. Types de recuperation

Recuperation automatique d'instance (Instance Recovery)

Comment ca marche: Se produit automatiquement au redemarrage apres un crash (SHUTDOWN ABORT, coupure de courant, panne materielle). Oracle detecte que les datafiles ne sont pas synchronises avec les redo logs (checkpoint incomplet).
Phase 1 — Roll Forward: Oracle rejoue tous les redo logs pour avancer les datafiles jusqu'au dernier etat enregistre.
Phase 2 — Rollback: Oracle annule les transactions non committees au moment du crash en utilisant les segments Undo.
Aucune commande DBA n'est requise. Tout est automatique lors du STARTUP. 
-- Aucune action requise: juste demarrer la base normalement
STARTUP;
-- Oracle detecte automatiquement le besoin de recuperation d'instance
-- et l'effectue avant d'ouvrir la base (phase MOUNT → OPEN)

Recuperation complete (Complete Recovery)

Comment ca marche: On restaure les fichiers depuis la sauvegarde (RESTORE = copie physique), puis on applique tous les archived redo logs disponibles pour remettre la base dans son etat le plus recent possible (jusqu'au dernier commit valide). La base retrouve son etat le plus recent → ouverture normale sans RESETLOGS. 
-- Cas 1: perte d'un tablespace non critique (base reste ouverte)
ALTER TABLESPACE users OFFLINE IMMEDIATE;

RESTORE TABLESPACE users;
RECOVER TABLESPACE users;

ALTER TABLESPACE users ONLINE;

-- Cas 2: perte du tablespace SYSTEM ou UNDO (base doit etre en MOUNT)
SHUTDOWN ABORT;
STARTUP MOUNT;

RESTORE DATABASE;
RECOVER DATABASE;

ALTER DATABASE OPEN;       -- Recuperation complete → OPEN sans RESETLOGS

Recuperation incomplete / PITR (Point-In-Time Recovery)

Comment ca marche: On s'arrete volontairement a un instant specifique dans le passe (avant une suppression accidentelle, une mise a jour erronee...). On utilise SET UNTIL pour definir le point cible: une date/heure ou un SCN. Apres la recuperation, la base doit imperativement etre ouverte avec OPEN RESETLOGS (cree une nouvelle incarnation de la base). 
-- PITR par timestamp (date et heure)
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;

RUN {
  SET UNTIL TIME "TO_DATE('2026-06-06 14:30:00','YYYY-MM-DD HH24:MI:SS')";
  RESTORE DATABASE;
  RECOVER DATABASE;
}

ALTER DATABASE OPEN RESETLOGS;    -- OBLIGATOIRE apres tout PITR

-- PITR par SCN (plus precis qu'une date)
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;

RUN {
  SET UNTIL SCN 1500000;
  RESTORE DATABASE;
  RECOVER DATABASE;
}

ALTER DATABASE OPEN RESETLOGS;

-- Trouver le SCN d'un instant passe
SELECT current_scn FROM v$database;
SELECT * FROM employees AS OF SCN 1499000;  -- Verifier l'etat avant incident
Quand ca echoue:
  • RESTORE echoue: aucune sauvegarde → LIST BACKUP pour verifier ce qui est disponible
  • RECOVER echoue: archived logs manquants pour la periode → RESTORE ARCHIVELOG depuis une sauvegarde d'abord
  • Oubli de RESETLOGS apres PITR → ORA-01589 au tentative d'OPEN → utiliser ALTER DATABASE OPEN RESETLOGS
  • Recuperation du control file perdu → demarrer en NOMOUNT, RESTORE CONTROLFILE FROM AUTOBACKUP, puis MOUNT

2e. Recovery Catalog (RC) — 3 etapes

Comment ca marche: Le Recovery Catalog est une base Oracle distincte qui stocke les metadonnees RMAN: historique complet des sauvegardes, scripts RMAN stockes, informations sur toutes les bases enregistrees. Sans lui, RMAN utilise uniquement le control file (historique limite a sa taille). Avec lui: historique illimite, scripts partages entre bases, gestion multi-cibles.
Comparaison Control file vs Recovery Catalog:
Control file seul: simple, pas de base supplementaire, mais historique limite et pas de scripts stockes.
Recovery Catalog: historique complet, scripts RMAN stockes, gestion multi-bases, supporte KEEP FOREVER — mais necessite une base Oracle supplementaire a administrer.

Etape 1 — Creer l'utilisateur sur la base catalog

-- Sur la base qui va heberger le catalog (pas la base cible)
CREATE USER rcat_owner IDENTIFIED BY password
  DEFAULT TABLESPACE rcat_ts
  QUOTA UNLIMITED ON rcat_ts;

-- Accorder le role necessaire
GRANT RECOVERY_CATALOG_OWNER TO rcat_owner;
GRANT CREATE SESSION TO rcat_owner;

Etape 2 — Creer le schema du catalog (depuis RMAN)

-- Connexion RMAN avec base cible ET catalog
rman TARGET / CATALOG rcat_owner/password@catdb

-- Creer le schema du catalog (une seule fois)
CREATE CATALOG;

Etape 3 — Enregistrer la base cible

-- Enregistrer la base cible dans le catalog
REGISTER DATABASE;

-- Verifier l'enregistrement
LIST DB_UNIQUE_NAME ALL;

-- Synchroniser le catalog apres modification de structure
RESYNC CATALOG;

-- Supprimer le catalog (si necessaire)
DROP CATALOG;
Quand ca ne marche pas:
  • CREATE CATALOG echoue: RECOVERY_CATALOG_OWNER pas accorde → accorder le role d'abord
  • REGISTER DATABASE echoue: base cible non accessible → verifier la connexion TARGET
  • Catalog desynchronise: faire RESYNC CATALOG apres ajout de datafile ou modification de structure

2f. Script de restauration et recuperation

Script 1 — Recuperation complete de la base 
-- 1. Arreter la base
SHUTDOWN ABORT;

-- 2. Monter la base
STARTUP MOUNT;

-- 3. Verifier les sauvegardes disponibles
LIST BACKUP SUMMARY;

-- 4. Restaurer (copie physique depuis la sauvegarde)
RESTORE DATABASE;

-- 5. Recuperer (appliquer les archived logs)
RECOVER DATABASE;

-- 6. Ouvrir la base (recuperation complete → OPEN sans RESETLOGS)
ALTER DATABASE OPEN;
Script 2 — PITR par date (recuperation incomplete) 
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;

RUN {
  SET UNTIL TIME "TO_DATE('2026-06-06 14:30:00','YYYY-MM-DD HH24:MI:SS')";
  RESTORE DATABASE;
  RECOVER DATABASE;
}

ALTER DATABASE OPEN RESETLOGS;   -- TOUJOURS RESETLOGS apres SET UNTIL
Script 3 — Recuperation du control file perdu 
-- Control file perdu: demarrer en NOMOUNT uniquement
STARTUP NOMOUNT;

RESTORE CONTROLFILE FROM AUTOBACKUP;

-- Monter la base (control file restaure)
ALTER DATABASE MOUNT;

RESTORE DATABASE;
RECOVER DATABASE;

ALTER DATABASE OPEN RESETLOGS;
Script 4 — Recuperation d'un tablespace non critique 
-- Base reste ouverte: mettre le tablespace offline
ALTER TABLESPACE users OFFLINE IMMEDIATE;

RESTORE TABLESPACE users;
RECOVER TABLESPACE users;

ALTER TABLESPACE users ONLINE;
Regle absolue de sequence: SHUTDOWN → MOUNT → RESTORE → RECOVER → OPEN.
Jamais RECOVER avant RESTORE. Jamais OPEN avant RECOVER.
Apres tout SET UNTIL (PITR): OPEN RESETLOGS obligatoire, jamais OPEN seul.
Perte du control file: commencer en NOMOUNT (pas MOUNT).
Logique exercice 2: RMAN connait la structure Oracle → connexion avec TARGET → sauvegarde (whole/full/incremental) → en cas de panne: RESTORE (copie) puis RECOVER (applique les logs) → OPEN ou OPEN RESETLOGS selon le type de recuperation.
Exercice 3

Flashback Technology — types, fonctionnalites, roles, commandes

Flashback permet de retrouver des etats passes sans restauration de sauvegarde, en utilisant les segments Undo (pour les donnees) et les flashback logs (pour la base entiere).

Vue d'ensemble — les 6 types Flashback

Mecanismes fondamentaux: La plupart des Flashback utilisent les segments Undo (Oracle conserve les versions anterieures des donnees pendant UNDO_RETENTION secondes). Flashback Database utilise en plus les flashback logs (images des blocs avant modification, stockees dans la FRA). Flashback Drop utilise la corbeille (RECYCLEBIN) — aucun Undo requis.
TypeGranularitePrerequis cleModifie les donnees?
Flashback QueryLignes a un instant TUNDO_RETENTION suffisantNon — lecture seule
Version QueryHistorique complet d'une ligneUNDO_RETENTION suffisantNon — lecture seule
Transaction QueryToutes ops d'une transaction + SQL d'annulationUNDO_RETENTION + Supplemental LoggingNon
Flashback TableUne ou plusieurs tablesUNDO_RETENTION + ROW MOVEMENT actifOui
Flashback DropTable supprimee (corbeille)Corbeille non purgeeOui

1. Flashback Query (AS OF)

Role: Voir les donnees telles qu'elles existaient a un instant T dans le passe. Utile pour retrouver des donnees avant une mise a jour ou suppression accidentelle. La requete est en lecture seule: elle ne change rien dans la base. On peut ensuite utiliser le resultat pour faire un INSERT et reinjecter les donnees perdues. 
-- Par timestamp
SELECT * FROM employees
  AS OF TIMESTAMP TO_TIMESTAMP('2026-06-06 13:00:00', 'YYYY-MM-DD HH24:MI:SS');

-- Par SCN
SELECT * FROM employees AS OF SCN 1498000;

-- Reinjecter des lignes supprimees accidentellement
INSERT INTO employees
  SELECT * FROM employees
  AS OF TIMESTAMP (SYSTIMESTAMP - INTERVAL '2' HOUR)
  WHERE employee_id = 1001;
Quand ca echoue — ORA-01555 (snapshot too old): Les donnees Undo necessaires ont ete ecrases par d'autres transactions (UNDO_RETENTION trop court). → Augmenter la retention: ALTER SYSTEM SET undo_retention = 10800; (en secondes — ici 3 heures) → Ou garantir la retention: ALTER TABLESPACE undotbs1 RETENTION GUARANTEE;
Verification: SHOW PARAMETER undo_retention;

2. Flashback Version Query (VERSIONS BETWEEN)

Role: Voir toutes les versions successives d'une ligne entre deux instants. Chaque modification = une version. On recupere: qui a modifie, quand, quel type (I/U/D), et l'identifiant de transaction (XID) — utile pour la Transaction Query ensuite. 
-- Par timestamps
SELECT versions_starttime, versions_endtime,
       versions_operation, versions_xid,
       employee_id, salary
FROM employees
  VERSIONS BETWEEN TIMESTAMP
    TO_TIMESTAMP('2026-06-06 12:00:00','YYYY-MM-DD HH24:MI:SS')
    AND
    TO_TIMESTAMP('2026-06-06 15:00:00','YYYY-MM-DD HH24:MI:SS')
WHERE employee_id = 100;

-- Par SCN
SELECT versions_startscn, versions_endscn, versions_operation, salary
FROM employees
  VERSIONS BETWEEN SCN 1400000 AND 1600000
WHERE employee_id = 100;
-- Pseudo-colonnes disponibles avec VERSIONS BETWEEN
VERSIONS_STARTTIME   -- Debut de cette version (timestamp)
VERSIONS_ENDTIME     -- Fin de cette version
VERSIONS_STARTSCN    -- SCN de debut
VERSIONS_ENDSCN      -- SCN de fin
VERSIONS_OPERATION   -- I=INSERT  U=UPDATE  D=DELETE
VERSIONS_XID         -- Transaction ID (a utiliser avec Transaction Query)

3. Flashback Transaction Query

Role: Retrouver toutes les modifications d'une transaction specifique et obtenir le SQL inverse (UNDO_SQL) pour l'annuler manuellement. Workflow typique: 1) identifier le XID via Version Query → 2) interroger FLASHBACK_TRANSACTION_QUERY → 3) executer UNDO_SQL. Prerequis: Supplemental Logging active. 
-- Activer le Supplemental Logging (prerequis — a faire une seule fois)
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;

-- Interroger la vue avec le XID recupere depuis VERSIONS_XID
SELECT xid, operation, start_timestamp, commit_timestamp,
       logon_user, undo_sql
FROM flashback_transaction_query
WHERE xid = HEXTORAW('0002000300000076')
ORDER BY seq DESC;

-- Executer ce SQL pour annuler manuellement la transaction

4. Flashback Table

Role: Remettre une ou plusieurs tables dans leur etat a un instant passe, sans restauration de sauvegarde. Modifie reellement les donnees. Prerequis obligatoire: ROW MOVEMENT active sur la table. Sans cela, Oracle ne peut pas repositionner les lignes lors du flashback. 
-- Prerequis: activer ROW MOVEMENT
ALTER TABLE employees ENABLE ROW MOVEMENT;

-- Flashback par timestamp
FLASHBACK TABLE employees
  TO TIMESTAMP TO_TIMESTAMP('2026-06-06 13:00:00','YYYY-MM-DD HH24:MI:SS');

-- Flashback par SCN
FLASHBACK TABLE employees TO SCN 1498000;

-- Plusieurs tables simultanement (coheference referentielle preservee)
FLASHBACK TABLE employees, departments
  TO TIMESTAMP TO_TIMESTAMP('2026-06-06 13:00:00','YYYY-MM-DD HH24:MI:SS');

-- Vers un restore point nomme
FLASHBACK TABLE employees TO RESTORE POINT avant_maj;
Quand ca echoue: ORA-08189: ROW MOVEMENT non actif → ALTER TABLE ... ENABLE ROW MOVEMENT
ORA-01555: periode trop ancienne → augmenter UNDO_RETENTION
Contraintes d'integrite bloquent: flashback toutes les tables liees ensemble simultanement

5. Flashback Drop (Corbeille)

Role: Recuperer une table supprimee par erreur. Oracle ne supprime pas immediatement les tables avec DROP TABLE: il les renomme en BIN$... et les conserve dans la RECYCLEBIN. Flashback Drop restaure la table depuis la corbeille. Echoue si: la table a ete supprimee avec DROP TABLE ... PURGE. 
-- Voir le contenu de la corbeille
SELECT object_name, original_name, type, droptime FROM recyclebin ORDER BY droptime DESC;
SELECT owner, object_name, original_name FROM dba_recyclebin;  -- Toutes les corbeilles (DBA)

-- Restaurer sous le nom d'origine
FLASHBACK TABLE employees TO BEFORE DROP;

-- Restaurer en renommant
FLASHBACK TABLE employees TO BEFORE DROP RENAME TO employees_restored;

-- Si plusieurs versions du meme nom dans la corbeille: utiliser le nom BIN$
FLASHBACK TABLE "BIN$xqTpY+9bQdugQKjH0rHKhA==$0" TO BEFORE DROP;

-- Gestion de la corbeille
PURGE RECYCLEBIN;                 -- Vider la corbeille de la session
PURGE TABLE employees;            -- Supprimer une table specifique de la corbeille
DROP TABLE employees PURGE;       -- Supprimer sans passer par la corbeille

ALTER SYSTEM SET recyclebin = ON;   -- Activer la corbeille (defaut ON)
SHOW PARAMETER recyclebin;

6. Flashback Database

Role: Revenir a un etat passe de toute la base. Beaucoup plus rapide qu'un PITR RMAN car utilise les flashback logs (images des blocs avant modification, dans la FRA) au lieu de restaurer une sauvegarde complete. Prerequis: mode ARCHIVELOG + Flashback Database ON + FRA configuree.
Activer Flashback Database (prerequis — une seule fois) 
-- Configurer la FRA si pas deja fait
ALTER SYSTEM SET db_recovery_file_dest = '/u01/app/oracle/fra';
ALTER SYSTEM SET db_recovery_file_dest_size = 20G;

-- Activer Flashback Database (base doit etre en MOUNT)
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE FLASHBACK ON;
ALTER DATABASE OPEN;

-- Verifier
SELECT flashback_on FROM v$database;

-- Configurer la retention des flashback logs (defaut 1440 min = 24h)
ALTER SYSTEM SET db_flashback_retention_target = 2880;  -- 48 heures
Utiliser Flashback Database 
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;

-- Par timestamp
FLASHBACK DATABASE TO TIMESTAMP
  TO_TIMESTAMP('2026-06-06 12:00:00','YYYY-MM-DD HH24:MI:SS');

-- Ou par SCN
FLASHBACK DATABASE TO SCN 1400000;

-- Ou vers un restore point nomme
FLASHBACK DATABASE TO RESTORE POINT avant_migration;

-- Ouvrir la base avec RESETLOGS (obligatoire)
ALTER DATABASE OPEN RESETLOGS;
Quand ca ne marche pas:
  • FLASHBACK DATABASE OFF → activer avec ALTER DATABASE FLASHBACK ON en mode MOUNT
  • FRA non configuree → definir DB_RECOVERY_FILE_DEST et DB_RECOVERY_FILE_DEST_SIZE
  • Flashback logs insuffisants (point trop ancien) → augmenter DB_FLASHBACK_RETENTION_TARGET
  • Base en NOARCHIVELOG → passer en mode ARCHIVELOG d'abord

7. Restore Points

Role: Un restore point est un alias nomme associe a un SCN precis. Permet de marquer un point avant une operation risquee et d'y revenir avec un nom clair.

Restore point normal: valide tant que les donnees Undo ou flashback logs necessaires sont disponibles.
Guaranteed Restore Point: Oracle garantit que les flashback logs ne seront jamais ecrases. Toujours disponible jusqu'a suppression explicite — mais remplit progressivement la FRA. 
-- Creer un restore point normal
CREATE RESTORE POINT avant_migration;

-- Creer un guaranteed restore point (necessite Flashback Database ON)
CREATE RESTORE POINT avant_migration GUARANTEE FLASHBACK DATABASE;

-- Voir les restore points disponibles
SELECT name, scn, time, guarantee_flashback_database FROM v$restore_point;

-- Utiliser pour Flashback Database
FLASHBACK DATABASE TO RESTORE POINT avant_migration;

-- Utiliser pour Flashback Table
FLASHBACK TABLE employees TO RESTORE POINT avant_migration;

-- Supprimer apres usage (IMPORTANT — libere l'espace FRA)
DROP RESTORE POINT avant_migration;
Guaranteed Restore Point et espace FRA: Tant qu'il existe, Oracle ne peut pas ecraser les flashback logs associes. Si on oublie de le supprimer, la FRA se remplit progressivement et bloque les sauvegardes. → Toujours supprimer le guaranteed restore point apres que l'operation risquee s'est bien passee.
Logique exercice 3: Flashback = recuperation sans restauration. Undo → Query/Version/Transaction/Table. Corbeille → Drop. Flashback logs → Database. Plus la granularite est large (base entiere), plus les prerequis sont importants (ARCHIVELOG + FLASHBACK ON + FRA).
Resume des prerequis critiques par type Flashback: Query/Version → UNDO_RETENTION suffisant. Transaction → + Supplemental Logging (ALTER DATABASE ADD SUPPLEMENTAL LOG DATA). Table → + ENABLE ROW MOVEMENT sur la table. Drop → corbeille non purgee (pas de PURGE). Database → ARCHIVELOG + ALTER DATABASE FLASHBACK ON + FRA configuree.
