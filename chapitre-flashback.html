Vue d'ensemble — comparaison des fonctionnalites

La technologie Flashback Oracle exploite les segments Undo pour retrouver des etats passes des donnees, sans restauration de sauvegarde. Flashback Database utilise en plus les flashback logs.

1. Flashback Query (AS OF)

Interroge les donnees telles qu'elles existaient a un moment passe (timestamp ou SCN). Basee sur les donnees Undo. La requete est en lecture seule: elle ne modifie pas les donnees actuelles.

Syntaxe 
-- Par timestamp
SELECT * FROM employees
  AS OF TIMESTAMP TO_TIMESTAMP('2026-06-06 13:00:00', 'YYYY-MM-DD HH24:MI:SS');

-- Par SCN
SELECT * FROM employees
  AS OF SCN 1498000;

-- Retrouver des lignes supprimees accidentellement
SELECT * FROM orders
  AS OF TIMESTAMP (SYSTIMESTAMP - INTERVAL '2' HOUR)
WHERE order_id = 1001;
Parametres Undo a verifier 
-- Verifier la retention Undo configuree
SHOW PARAMETER undo_retention;

-- Verifier le tablespace Undo
SHOW PARAMETER undo_tablespace;

-- La periode de retention en secondes (900 = 15 minutes par defaut)
SELECT value FROM v$parameter WHERE name = 'undo_retention';
Limite: Flashback Query ne fonctionne que si les donnees Undo sont encore disponibles dans le tablespace UNDO. Si UNDO_RETENTION est trop court ou que l'espace Undo a ete reutilise, la requete echoue avec ORA-01555 (snapshot too old).

2. Flashback Version Query (VERSIONS BETWEEN)

Montre toutes les versions successives d'une ligne entre deux instants. Retourne une ligne par modification de la ligne dans la periode donnee. Utile pour comprendre l'historique complet des modifications.

Syntaxe 
-- Historique d'une ligne entre deux timestamps
SELECT versions_starttime, versions_endtime,
       versions_operation, versions_xid,
       employee_id, salary
FROM employees
  VERSIONS BETWEEN TIMESTAMP
    TO_TIMESTAMP('2026-06-06 12:00:00','YYYY-MM-DD HH24:MI:SS')
    AND
    TO_TIMESTAMP('2026-06-06 15:00:00','YYYY-MM-DD HH24:MI:SS')
WHERE employee_id = 100;

-- Par SCN
SELECT versions_startscn, versions_endscn, versions_operation, salary
FROM employees
  VERSIONS BETWEEN SCN 1400000 AND 1600000
WHERE employee_id = 100;
Pseudo-colonnes disponibles 
VERSIONS_STARTTIME   -- Timestamp de debut de cette version
VERSIONS_ENDTIME     -- Timestamp de fin de cette version
VERSIONS_STARTSCN    -- SCN de debut
VERSIONS_ENDSCN      -- SCN de fin
VERSIONS_OPERATION   -- I (insert), U (update), D (delete)
VERSIONS_XID         -- Transaction ID

3. Flashback Transaction Query

Retrouve toutes les modifications effectuees par une transaction specifique. Fournit le SQL d'annulation (UNDO_SQL) pour annuler selectivement une transaction. Necessite que le Supplemental Logging soit active.

Activer le Supplemental Logging 
-- Activer le Supplemental Logging minimal (necessaire pour UNDO_SQL)
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;
Interroger FLASHBACK_TRANSACTION_QUERY 
-- Voir toutes les operations d'une transaction via son XID
SELECT xid, operation, start_timestamp, commit_timestamp,
       logon_user, undo_sql
FROM flashback_transaction_query
WHERE xid = HEXTORAW('0002000300000076');

-- Trouver le XID d'une transaction via Flashback Version Query
-- (utiliser VERSIONS_XID de la requete VERSIONS BETWEEN)

-- Afficher le SQL d'annulation genere automatiquement
SELECT undo_sql FROM flashback_transaction_query
WHERE xid = HEXTORAW('0002000300000076')
ORDER BY seq DESC;
Usage typique: 1) Identifier la transaction fautive via VERSIONS BETWEEN (recuperer VERSIONS_XID). 2) Interroger FLASHBACK_TRANSACTION_QUERY avec ce XID pour obtenir le SQL d'annulation. 3) Executer le SQL d'annulation pour corriger les donnees.

4. Flashback Table

Remet une ou plusieurs tables dans leur etat a un instant passe (timestamp ou SCN). Modifie reellement les donnees de la table. La table doit avoir ROW MOVEMENT active.

Prerequis et preparation 
-- Activer ROW MOVEMENT sur la table (prerequis obligatoire)
ALTER TABLE employees ENABLE ROW MOVEMENT;

-- Verifier
SELECT row_movement FROM dba_tables WHERE table_name = 'EMPLOYEES';
Syntaxe Flashback Table 
-- Flashback par timestamp
FLASHBACK TABLE employees
  TO TIMESTAMP TO_TIMESTAMP('2026-06-06 13:00:00','YYYY-MM-DD HH24:MI:SS');

-- Flashback par SCN
FLASHBACK TABLE employees TO SCN 1498000;

-- Flashback de plusieurs tables simultanement
FLASHBACK TABLE employees, departments
  TO TIMESTAMP TO_TIMESTAMP('2026-06-06 13:00:00','YYYY-MM-DD HH24:MI:SS');

-- Flashback vers un restore point nomme
FLASHBACK TABLE employees TO RESTORE POINT avant_maj;
Important: FLASHBACK TABLE modifie les donnees actuelles. Sauvegarder ou noter l'etat courant avant d'executer si necessaire. L'operation est irreversible sans une autre Flashback Table vers l'etat precedent.

5. Flashback Drop (Corbeille)

Oracle ne supprime pas immediatement les tables avec DROP TABLE: il les place dans la corbeille (RECYCLEBIN). Flashback Drop permet de les restaurer.

Verifier et lister la corbeille 
-- Verifier le contenu de la corbeille (objets supprimes)
SELECT object_name, original_name, type, droptime
FROM recyclebin
ORDER BY droptime DESC;

-- Ou via DBA_RECYCLEBIN (pour les administrateurs, toutes les corbeilles)
SELECT owner, object_name, original_name, type, droptime
FROM dba_recyclebin;
Restaurer une table depuis la corbeille 
-- Restaurer sous le nom d'origine
FLASHBACK TABLE employees TO BEFORE DROP;

-- Restaurer en renommant
FLASHBACK TABLE employees TO BEFORE DROP RENAME TO employees_restored;

-- Si plusieurs versions du meme nom existent dans la corbeille :
-- utiliser le nom de la corbeille (objet renomme en BIN$...)
FLASHBACK TABLE "BIN$xqTpY+9bQdugQKjH0rHKhA==$0" TO BEFORE DROP;
Gestion de la corbeille 
-- Vider la corbeille de la session courante
PURGE RECYCLEBIN;

-- Vider une table specifique de la corbeille
PURGE TABLE employees;

-- DROP TABLE sans passer par la corbeille (purge immediate)
DROP TABLE employees PURGE;

-- Activer/desactiver la corbeille au niveau systeme
ALTER SYSTEM SET recyclebin = ON;
ALTER SYSTEM SET recyclebin = OFF;

-- Verifier
SHOW PARAMETER recyclebin;

6. Flashback Database

Revient a un etat passe de toute la base, similaire a un PITR mais beaucoup plus rapide car utilise les flashback logs au lieu des archived logs. Necessite l'activation prealable et une FRA configuree.

Activer Flashback Database 
-- Prerequis : FRA configuree et mode ARCHIVELOG
ALTER SYSTEM SET db_recovery_file_dest = '/u01/app/oracle/fra';
ALTER SYSTEM SET db_recovery_file_dest_size = 20G;

-- Activer Flashback Database (base doit etre en MOUNT)
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE FLASHBACK ON;
ALTER DATABASE OPEN;

-- Verifier
SELECT flashback_on FROM v$database;

-- Configurer la retention des flashback logs (en minutes, defaut 1440 = 24h)
ALTER SYSTEM SET db_flashback_retention_target = 2880;  -- 48 heures
Utiliser Flashback Database 
-- 1. Mettre la base en MOUNT
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;

-- 2. Flashback vers un timestamp
FLASHBACK DATABASE TO TIMESTAMP
  TO_TIMESTAMP('2026-06-06 12:00:00','YYYY-MM-DD HH24:MI:SS');

-- Ou vers un SCN
FLASHBACK DATABASE TO SCN 1400000;

-- Ou vers un restore point nomme
FLASHBACK DATABASE TO RESTORE POINT avant_migration;

-- 3. Ouvrir la base avec RESETLOGS (obligatoire)
ALTER DATABASE OPEN RESETLOGS;
Avantage vs PITR RMAN: Flashback Database est beaucoup plus rapide car il rejoue les flashback logs (journaux des blocs avant modification) au lieu de restaurer toute une sauvegarde et de rejouer des archives. Ideal pour les retours rapides.

7. Restore Points

Un restore point est un alias nomme associe a un SCN precis. Permet de designer un point de retour par un nom significatif au lieu d'un timestamp ou SCN numerique. Deux types: normal et garanti.

TypeDescriptionRetention
Restore Point normalAlias d'un SCN. Valide tant que les donnees Undo ou flashback logs necessaires sont disponibles.Limitee (depend de UNDO_RETENTION et flashback logs).
Guaranteed Restore PointOracle garantit que les flashback logs necessaires ne seront pas ecrases. Toujours disponible.Permanente jusqu'a suppression explicite.
Creer et utiliser des Restore Points 
-- Creer un restore point normal
CREATE RESTORE POINT avant_migration;

-- Creer un guaranteed restore point (necessite Flashback Database actif)
CREATE RESTORE POINT avant_migration GUARANTEE FLASHBACK DATABASE;

-- Voir les restore points disponibles
SELECT name, scn, time, guarantee_flashback_database
FROM v$restore_point;

-- Utiliser un restore point pour Flashback Database
FLASHBACK DATABASE TO RESTORE POINT avant_migration;

-- Utiliser un restore point pour Flashback Table
FLASHBACK TABLE employees TO RESTORE POINT avant_migration;

-- Supprimer un restore point
DROP RESTORE POINT avant_migration;
Garanteed Restore Point: Puisque les flashback logs ne peuvent pas etre ecrases, un Guaranteed Restore Point peut remplir l'espace FRA si on oublie de le supprimer. Toujours le supprimer apres usage.
