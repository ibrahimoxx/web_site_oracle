Sauvegarde et recuperation

Pannes, restore vs recover, instance recovery, ARCHIVELOG, FRA, control files, redo logs — tout le vocabulaire central avec les commandes exactes.

Role DBA Pannes Restore vs Recover Instance recovery ARCHIVELOG FRA Fichiers critiques

1. Responsabilites du DBA

  • MTBF (Mean Time Between Failures) — temps moyen entre deux pannes. Plus il est eleve, plus le systeme est fiable. Le DBA doit l'augmenter.
  • MTTR (Mean Time To Recover) — temps moyen de recuperation. Plus il est faible, plus la reprise est rapide. Le DBA doit le reduire.
  • Garantir la disponibilite de la base quand les utilisateurs en ont besoin.
  • Anticiper les pannes et preparer les procedures de reprise.
  • Minimiser la perte de donnees, idealement zero transaction validee perdue.

Formule a retenir

MTBF eleve = pannes rares.
MTTR faible = reprise rapide.
L'objectif du DBA: les deux en meme temps.

2. Categories de pannes

Type de panneExempleSolution
Echec d'instructionINSERT invalide, manque de privileges, espace insuffisantCorriger la requete, accorder les droits, ajuster les quotas
Echec processus utilisateurSession interrompue brutalementPMON nettoie automatiquement les transactions non validees et libere les ressources

Panne physiqueDisque HS, datafile corrompu ou supprimeRestaurer le fichier depuis une sauvegarde, puis recuperer

3. Restauration vs Recuperation

Ne jamais confondre ces deux operations. Elles sont complementaires mais distinctes.
  • RESTORE (restauration) — recopier un fichier de sauvegarde a son emplacement. Oracle peut de nouveau le lire, mais il n'est pas forcement a jour.
  • RECOVER (recuperation) — appliquer les redo logs et archives pour remettre le fichier restaure dans un etat coherent et a jour.
La recuperation suit toujours la restauration. Restaurer remet le support; recuperer remet le contenu a jour.

Portee de la recuperation

TypeDefinitionCommande finale
CompleteRevenir a l'etat le plus recent possible (toutes les transactions validees).ALTER DATABASE OPEN;
Incomplete (PITR)Revenir a un point precis: heure, SCN ou sequence.ALTER DATABASE OPEN RESETLOGS;

4. Recuperation automatique de l'instance

Si l'instance s'arrete brutalement (panne courant, crash processus), les datafiles ne sont pas synchronises avec les redo logs. Au redemarrage, Oracle lance une recuperation automatique en deux phases:

PhaseNomAction
1Roll-forwardOracle rejoue toutes les modifications enregistrees dans les redo logs (validees ou non).
2RollbackOracle annule les transactions non validees en utilisant les segments Undo.
Cette reprise est automatique et transparente. L'administrateur n'a rien a faire. La base s'ouvre proprement apres les deux phases.

5. Mode ARCHIVELOG

ModeComportementRecuperation possible
NOARCHIVELOGLes redo logs sont reutilises sans etre archives. Historique perdu.Limitee: uniquement recuperation d'instance. Pas de recuperation complete apres panne physique.
ARCHIVELOGLes redo logs sont archives avant reutilisation. Historique complet conserve.Complete: recuperation jusqu'au dernier commit, sauvegarde a chaud possible.
Verifier le mode actuel 
ARCHIVE LOG LIST;

SELECT log_mode FROM v$database;
Activer le mode ARCHIVELOG 
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE ARCHIVELOG;
ALTER DATABASE OPEN;
Desactiver le mode ARCHIVELOG 
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE NOARCHIVELOG;
ALTER DATABASE OPEN;
Exam: La sequence en 4 etapes est obligatoire. La base doit etre en mode MOUNT (pas OPEN) pour changer le mode d'archivage.

6. Fast Recovery Area (FRA)

Zone de stockage Oracle reservee aux fichiers de recuperation. Elle centralise: pieces de sauvegarde, copies, archived redo logs, control file autobackups, flashback logs.

Parametres de configuration

ParametreRole
DB_RECOVERY_FILE_DESTChemin du repertoire FRA.
DB_RECOVERY_FILE_DEST_SIZETaille maximale allouee a la FRA.
Configurer la FRA 
ALTER SYSTEM SET db_recovery_file_dest = '/u01/app/oracle/fra';
ALTER SYSTEM SET db_recovery_file_dest_size = 20G;
Verifier l'utilisation de la FRA 
-- Taille et espace utilise
SELECT name, space_limit, space_used, space_reclaimable, number_of_files
  FROM v$recovery_file_dest;

-- Utilisation par type de fichier
SELECT file_type, percent_space_used, percent_space_reclaimable, number_of_files
  FROM v$recovery_area_usage;

7. Fichiers critiques — Control files et Redo logs

Control files

Contiennent les metadonnees de la base: structure, datafiles, redo logs, checkpoints. Oracle recommande de les multiplexer (plusieurs copies sur disques differents).

Gestion des control files 
-- Voir la liste des control files
SHOW PARAMETER control_files;

SELECT name FROM v$controlfile;

Redo logs

Enregistrent toutes les modifications. Utilises pour la recuperation. Doivent etre multiplexes (plusieurs membres par groupe sur disques differents).

Gestion des redo logs 
-- Voir les groupes de redo logs
SELECT group#, sequence#, status, members, bytes
  FROM v$log;

-- Voir les membres de chaque groupe
SELECT group#, member, status
  FROM v$logfile
  ORDER BY group#;

-- Ajouter un membre a un groupe existant (multiplexage)
ALTER DATABASE ADD LOGFILE MEMBER '/u02/oradata/redo01b.log' TO GROUP 1;

-- Ajouter un nouveau groupe
ALTER DATABASE ADD LOGFILE GROUP 4 ('/u01/oradata/redo04.log') SIZE 200M;

Commandes generales de recuperation

Sequence de recuperation complete 
STARTUP;
STARTUP MOUNT;
SHUTDOWN IMMEDIATE;

RESTORE DATABASE;
RECOVER DATABASE;

RESTORE TABLESPACE nom_tablespace;
RECOVER TABLESPACE nom_tablespace;

RESTORE DATAFILE numero_datafile;
RECOVER DATAFILE numero_datafile;

-- Ouvrir apres recuperation complete
ALTER DATABASE OPEN;

-- Ouvrir apres recuperation incomplete (nouvelle incarnation)
ALTER DATABASE OPEN RESETLOGS;
