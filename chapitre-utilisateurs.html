Gestion des utilisateurs, des privileges, des roles et des profils

Ce chapitre traite toute la securite logique de base dans Oracle: qui entre dans la base, avec quels droits, dans quel espace de travail, et avec quelles contraintes de mot de passe et de ressources.

Idee cle Utilisateurs Privileges Roles Profils

Idee cle du chapitre

Oracle etant un SGBD multiutilisateur, la securite ne se limite pas a bloquer l’acces. Il faut aussi organiser les droits de facon fine. Un utilisateur doit pouvoir se connecter, creer ou lire uniquement ce qui lui est permis, consommer un espace controle et respecter une politique de mot de passe.

Un bon administrateur ne donne pas tous les droits a tout le monde. Il affecte le minimum necessaire, puis structure les autorisations pour qu’elles restent simples a maintenir.

Ce qu’il faut absolument retenir

  • Utilisateur = identite Oracle.
  • Privilege systeme = action generale sur la base.
  • Privilege objet = action sur un objet precis.
  • Role = groupe de privileges.
  • Profil = limites de mot de passe et de ressources.

1. Gestion des utilisateurs

Ce qu’on decide lors de la creation

  • Le nom de l’utilisateur.
  • La methode d’authentification.
  • Le tablespace par defaut et le tablespace temporaire.
  • Les quotas sur les tablespaces.
  • Les roles et privileges de depart.

Authentification

  • Par la base de donnees: mode le plus courant, Oracle verifie l’identifiant et le mot de passe.
  • Par le systeme d’exploitation: Oracle s’appuie sur l’authentification du systeme.

Operations essentielles sur l’utilisateur

  • Creation avec CREATE USER.
  • Modification avec ALTER USER.
  • Suppression avec DROP USER, eventuellement avec suppression du schema.
  • Consultation des informations avec des vues comme DBA_USERS.
Le cours insiste sur le fait qu’un utilisateur ne se resume pas a un login. Il a aussi un ancrage dans l’espace de stockage via les tablespaces et les quotas.

2. Gestion des privileges

Les privileges servent a securiser l’acces aux ressources Oracle. Le chapitre distingue clairement deux familles.

Type Definition Exemples
Privilege systeme Autorise une action generale sur la base ou sur des categories d’objets. CREATE SESSION, CREATE TABLE, CREATE TABLESPACE
Privilege objet Autorise une action sur un objet donne appartenant a un schema. SELECT, INSERT, UPDATE, DELETE sur une table

Gestion pratique

  • Accorder avec GRANT.
  • Retirer avec REVOKE.
  • Le privilege peut etre direct ou transmis via un role.
  • Pour les privileges objet, on peut parfois accorder avec WITH GRANT OPTION selon le contexte.

3. Gestion des roles

Un role est un regroupement nomme de privileges. L’interet est administratif: au lieu de distribuer les memes privileges un par un a plusieurs utilisateurs, on les met dans un role puis on attribue ce role.

  • Creation d’un role et ajout de privileges.
  • Attribution du role aux utilisateurs.
  • Consultation des roles et des privileges associes via le dictionnaire.
  • Suppression du role si necessaire.

Roles standards cites par le cours

  • CONNECT
  • RESOURCE
  • DBA

Le cours les presente comme roles systeme standards a connaitre.

4. Gestion des profils et des mots de passe

Le profil est l’outil Oracle pour centraliser des limites relatives aux mots de passe et a certaines ressources systeme. Le cours le rattache directement a l’augmentation de la securite de la base.

Limites de mot de passe

  • FAILED_LOGIN_ATTEMPTS: nombre maximal d’essais de connexion echoues.
  • PASSWORD_LOCK_TIME: duree de verrouillage apres echecs.
  • PASSWORD_LIFE_TIME: duree de validite du mot de passe.

Limites de ressources

  • SESSIONS_PER_USER: nombre de sessions simultanees.
  • CONNECT_TIME ou limites proches: temps de connexion autorise.
  • LOGICAL_READS_PER_SESSION: volume de lectures logiques.

Demarche logique

  1. Creer le profil avec ses limites.
  2. L’affecter aux utilisateurs.
  3. Verifier les parametres via les vues du dictionnaire.

Resume flash

Dans ce chapitre, Oracle se comporte comme un environnement controle: les utilisateurs sont identifies, les actions sont autorisees ou refusees par privileges, les droits peuvent etre factorises par roles, et les politiques de securite sont durcies par profils.
