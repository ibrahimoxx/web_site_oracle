Restauration et recuperation avec RMAN

Cette partie montre comment RMAN est utilise quand la panne est deja la: fichiers critiques perdus, recuperation complete ou incomplete, control file de sauvegarde et spfile a restaurer.

1. Principe general

  • RESTORE remet en place les fichiers depuis la sauvegarde.
  • RECOVER applique les modifications necessaires via redo ou archives.
  • Le couple restore + recover constitue la logique de reprise complete.

2. Recuperation complete

L’objectif est de revenir a l’etat le plus a jour possible de la base en appliquant toutes les modifications valides disponibles.

  • Souvent utilise apres perte d’un datafile critique.
  • Peut necessiter un montage de la base avant la reprise.

3. Recuperation incomplete

On vise un point precis dans le temps, un SCN ou un moment donne. Elle sert quand on ne veut pas rejouer toutes les modifications ou quand certaines sont perdues ou inde desirables.

  • Elle est pilotee par un critere d’arret (UNTIL TIME, UNTIL SCN, etc.).
  • L’ouverture de la base apres une telle reprise impose souvent RESETLOGS.

4. Cas cites par le cours

  • Perte d’un fichier de donnees critique.
  • Recuperation a partir d’un fichier de controle de sauvegarde.
  • Restauration du spfile depuis une sauvegarde automatique.

5. Control file de sauvegarde

Si le control file restaure n’est pas a jour, il faut resynchroniser la base via une recuperation, puis ouvrir souvent avec RESETLOGS. Cela cree une nouvelle incarnation logique.

6. Spfile

Le cours montre que RMAN peut aider a restaurer le spfile, meme dans un contexte ou l’instance ne demarre pas normalement, en s’appuyant sur des sauvegardes automatiques et parfois sur le DBID.

Resume flash

L’idee centrale est la suivante: en recovery, RMAN doit retrouver les bons fichiers, les restaurer, les remettre a jour, puis rouvrir la base de maniere coherente. L’examen aime surtout les differences complete/incomplete et le role de RESETLOGS.
